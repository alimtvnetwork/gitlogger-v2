## Module Summary

This module defines strict standards for error casing (UrlError vs URLError) and mandates a 'Code Red' logging policy for all file/path errors, requiring specific metadata like exact paths and standardized failure reasons.

---

## Inlined Contracts

> Required artifacts inlined here so each AC is self-contained — a mediocre AI does not need to chase cross-links.

### Failure Reason Enums
| Value | Context |
|-------|---------|
| `FileDoesNotExist` | File confirmed absent |
| `PathInvalid` | Malformed/unresolvable/security escape |
| `PathInaccessible` | Network/mount access issues |
| `FileNameMismatch` | Casing or name difference |
| `ExtensionMismatch` | Wrong file extension |
| `PermissionDenied` | OS-level lock (0000 mode) |
| `FileNeverCreated` | Pipeline/Build failure |
| `FileMovedOrRenamed` | Existed previously, now gone |

### Approved Operations
`Read`, `Write`, `Copy`, `Move`, `Inject`, `Load`, `Extract`, `Resolve`

### Casing Convention
- NO: `URLError`, `WrapURLError`
- YES: `UrlError`, `WrapUrlError`
- Reason: Mixed-case identifier convention (Title Case acronyms)

### Mandatory Log Structure
- Level: Code Red (or equivalent)
- Fields: path, operation, reason, module, timestamp
- Optional: recovery action

## Acceptance Criteria

### AC-01: Strict Enforce of Failure Reason Enums  `[critical]`
- **Given** A Go implementation of a file read error constructor
- **When** The code creates a new apperror instance for a file/path failure
- **Then** The field 'reason' must be one of the enums defined in the Failures table (e.g., 'FileDoesNotExist', 'PathInvalid', 'FileNeverCreated')
- **Verifies:** error-management-file-path-and-missing-file-code-red-rule.md § 3

### AC-02: Mandatory Logging Fields Presence  `[critical]`
- **Given** An error log generated by the CSSInjector module
- **When** An asset load failure occurs (Operation: Load)
- **Then** The log MUST contain the 'Exact file path', 'Operation', 'Failure reason', and 'Module' fields as specified in the Mandatory Logging Fields table
- **Verifies:** error-management-file-path-and-missing-file-code-red-rule.md § 2

### AC-03: Restricted Operation Field Values  `[high]`
- **Given** A developer implementing file manipulation logic
- **When** Logging a system-wide file-related error
- **Then** The 'operation' field must be mapped to one of: Read, Write, Copy, Move, Inject, Load, Extract, or Resolve
- **Verifies:** error-management-file-path-and-missing-file-code-red-rule.md § 2.1

### AC-04: Consistency of UrlError Casing  `[medium]`
- **Given** The spec files 02-apperror-struct.md, readme.md, and 98-changelog.md
- **When** Performing a case-sensitive search across the spec tree
- **Then** Zero instances of the string 'URLError' or 'WrapURLError' shall exist; they must be 'UrlError' and 'WrapUrlError' respectively
- **Verifies:** 2026-04-02-url-error-casing-fix.md

### AC-05: TypeScript Implementation Alignment  `[high]`
- **Given** A TypeScript environment using captureError() for a missing plugin file
- **When** An archive is extracted but lacks the expected main.php entry
- **Then** The context object must include a 'reason' of 'FileDoesNotExist' and an 'operation' of 'Extract' as per the PluginInstaller example
- **Verifies:** error-management-file-path-and-missing-file-code-red-rule.md § 4.4 / § 5.2

### AC-06: Invalid Path Security Logging  `[critical]`
- **Given** A path resolution attempt that results in a traversal/escape attempt (e.g., /etc/passwd)
- **When** The AssetLoader module detects a path escaping the allowed directory
- **Then** The logged reason must be 'PathInvalid' and the operation must be 'Resolve'
- **Verifies:** error-management-file-path-and-missing-file-code-red-rule.md § 4.2
